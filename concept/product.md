# プロダクト仕様

## 製品の本質

> **AIエージェントの危険な実行を、人間承認・監査ログ・権限分離で本番運用できるようにする実行権限管理インフラ**

重要な点は、承認通知やSDKデコレータそのものではない。価値の中心は、AIエージェントの実行を一時停止し、業務担当者の判断を受け取り、その結果をエージェントに返し、あとから説明できる監査証跡を残すことにある。

SDKは導入手段の1つ。LangGraph、OpenAI Agents SDK、Microsoft系ツールが承認・interruptを内蔵し始めているため、稟議ゲート（仮）は「低レベルのpause/resume機構」ではなく、**業務運用・監査・権限分離レイヤー**として設計する。

## 機能方針

### Phase 0: 実装前検証

| 成果物 | 目的 |
|--------|------|
| 10社ヒアリング | 実際に止めたいAI実行、既存の自前承認、支払い意思を確認する |
| 薄いデモ | Slack承認とWeb監査ログを見せ、抽象論を避ける |
| LINE WORKS Waitlist | 国内企業での通知チャネル需要を実装前に測る |
| MVPスコープ確定 | 最初の言語、通知チャネル、Level 2対象ユースケースを決める |

### Phase 1: MVP

| 機能 | 概要 |
|------|------|
| SDK/Adapter（1言語から開始） | AIエージェントの危険なツール実行前に承認リクエストを作成する |
| Slack承認通知 | アクション内容・理由・主要メタデータをSlackに送信し、承認/却下できる |
| 管理ダッシュボード | 承認待ち一覧、処理履歴、監査ログ、プロジェクト/APIキー管理を提供する |
| 監査ログ | 誰が何をいつ承認/却下したかを変更不可の履歴として残す |
| fail-closed | 承認サービス障害時はデフォルト拒否で動作する |
| テストモード | 通知なしで承認フローを確認できるステージング/CI向け機能 |

MVPでは、複数承認、エスカレーション、モバイルアプリ、LINE WORKS本実装、汎用クレデンシャル保管は作らない。初期顧客が「これがないと使えない」と示したものだけを追加する。

### Phase 1.5: 国内導入強化

| 機能 | 採用条件 |
|------|----------|
| LINE WORKS連携 | 10社ヒアリングのうち3社以上が必須チャネルとして挙げる |
| Microsoft Teams連携 | Microsoft 365環境の顧客がSlackより多い場合に優先する |
| 承認ルールUI | 手動承認が多すぎて運用負荷が課題になった場合に実装する |

LINE WORKSは国内中堅企業への重要な入口だが、最初から作るのではなく、Waitlistとヒアリングで需要を確認してから実装する。

### Phase 2: Level 2 クレデンシャル保管

| 機能 | 概要 |
|------|------|
| 限定用途の代理実行 | まずはメール送信、支払い、CRM更新のうち1用途だけ対応する |
| クレデンシャル保管 | AIエージェントに本物のAPIキーを渡さず、稟議ゲート（仮）が保管する |
| 代理実行ログ | 承認から外部API実行までを監査ログに残す |
| 権限テンプレート | 「一斉メール」「高額支払い」など危険操作の標準ルールを提供する |

Phase 2は差別化の中核だが、技術・運用責任が重い。汎用Vaultとして始めず、1つの強いユースケースで構造的保証を証明する。

## アーキテクチャ

### Level 1 データフロー

```
[AIエージェント]
    │
    │ SDK/Adapterが危険なツール実行を検知
    ▼
[稟議ゲート（仮） API]
    ├── 承認リクエストを記録
    ├── Slackに通知
    └── Webダッシュボード/監査ログに反映
          │
          ▼
    [承認者が判断]
          │
          ▼
[AIエージェントへ承認/却下を返す]
```

### Level 2 データフロー

```
[AIエージェント]
    │ 実行依頼のみ送る（外部APIキーは持たない）
    ▼
[稟議ゲート（仮）]
    ├── 承認ルール/人間承認
    ├── クレデンシャルを安全に取得
    ├── 外部APIを代理実行
    └── 結果と監査ログを返す
```

## 技術スタック方針

| レイヤー | MVP方針 | 理由 |
|---------|---------|------|
| SDK/Adapter | PythonまたはTypeScriptの片方から開始 | 10社ヒアリングで利用比率を見て決める |
| バックエンド API | Supabase/PostgreSQL + Edge Functions または軽量APIサーバー | マルチテナント、RLS、低コストを優先 |
| 管理フロントエンド | React | ダッシュボードと監査ログUIを素早く作る |
| 通知インフラ | Slack APIをMVP、LINE WORKS/Teamsは検証後 | 実装量を抑えつつ通知抽象化は保つ |
| シークレット管理 | Phase 2でSupabase VaultまたはAWS Secrets Manager | Level 2まで実装しない限り不要 |

## セキュリティ設計

### 3段階のセキュリティレベル

#### Level 1: SDK/Adapter（導入容易性）

```python
@approval_required(reason="メール送信")
def send_email(...):
    ...
```

- 危険な関数やツール呼び出しの前に承認リクエストを作成する
- fail-closedで承認サービス障害時は拒否する
- 監査ログで「誰が何を承認したか」を残す
- 限界: デコレータの付け忘れ、削除、別経路実行は防げない

#### Level 2: クレデンシャル保管（構造的な保証）

```
現在:
  AIエージェント → SendGrid APIキーを保持 → 直接送信できる

Level 2:
  AIエージェント → 稟議ゲート（仮）に送信依頼 → 承認後に稟議ゲート（仮）が代理送信
```

AIに本物のAPIキーを渡さないことで、コードレベルのバイパスを難しくする。最初は「メール送信」など1用途で実装し、顧客価値と運用負荷を検証する。

#### Level 3: Outbound Proxy（将来）

- AIエージェントの実行環境をネットワークレベルで制限する
- 全アウトバウンド通信を稟議ゲート（仮）のProxyに強制経由する
- 顧客のインフラ設定変更が必要なため、初期スコープから外す

### セキュリティの優先順位

```
Phase 1:   Level 1 + fail-closed + 監査ログ
Phase 2:   限定用途のLevel 2クレデンシャル保管
Phase 3:   Enterprise向けLevel 3/コンプライアンス対応
```

## すり抜けリスクと対策

| リスク | 深刻度 | 対策 |
|--------|--------|------|
| デコレータの付け忘れ | 高 | MVPでは注意喚起とサンプル、Phase 2で根本対策 |
| AIが別ルートで同じ操作を実行 | 高 | Level 2でAPIキーをAIから分離する |
| ネイティブinterruptとの競合 | 高 | SDK機構ではなく承認UI・監査・権限管理を価値にする |
| 承認が面倒で使われない | 中 | ルール、自動承認、通知UXを顧客ヒアリング後に追加する |
| 承認サービス障害 | 中 | fail-closed、ステータス通知、テストモードを用意する |

## 対応エージェントフレームワーク方針

| フレームワーク | 方針 |
|--------------|------|
| LangGraph / LangChain | MVPで最初のサンプル対象。ネイティブinterruptを補完するUI/監査として接続する |
| OpenAI Agents SDK | 早期サンプル対象。tool approvalと共存するAdapterを用意する |
| CrewAI / AutoGen | 顧客需要が確認できた順にサンプル追加 |
| カスタム実装 | Webhook/REST APIで常時対応する |
