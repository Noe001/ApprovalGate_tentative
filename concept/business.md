# ビジネスモデル

## 事業仮説

稟議ゲート（仮）は、AIエージェントに実行権限を渡したい企業が、本番導入前に必要とする「人間承認・監査ログ・権限分離」を提供する。SDK導入の簡単さは入口だが、支払い理由は「事故を防ぐ」「社内稟議を通す」「顧客に説明できる」「AIにAPIキーを渡さない」にある。

---

## 顧客ペルソナ

### ペルソナA: AIスタートアップCTO

**状況**
- 受発注業務や顧客対応をAI自動化するSaaSの技術責任者
- AIが発注メール、CRM更新、支払い準備を自動化している
- 事故が怖く、最終実行だけ人間確認にしている

**今の対処法**
- エンジニアが自前のSlack承認Botを作った
- 監査ログ、権限管理、例外処理が弱く、メンテが属人化している

**刺さるポイント**
「自前承認Botのメンテをやめ、顧客に説明できる監査ログまで持てる」

### ペルソナB: 中堅SIer新規事業担当

**状況**
- AIエージェントを使った業務自動化サービスを提案している
- 顧客や社内稟議で「AIが勝手に動くのが怖い」と止められる

**今の対処法**
- 「人間が確認します」と口頭説明している
- 技術的な担保が弱く、PoCから本番に進まない

**刺さるポイント**
「承認画面と監査ログを見せて、AI本番導入の説明材料にできる」

### ペルソナC: AI機能を持つ小規模SaaS開発者

**状況**
- 顧客向けにAIメール送信、物件提案、営業代行、予約調整などを提供している
- 自動実行機能は価値が高いが、誤送信や誤更新が怖くて封印している

**今の対処法**
- AIは下書きまで、人間が手動実行
- 自動化の価値を十分に売れない

**刺さるポイント**
「顧客の担当者に承認通知を飛ばし、半自動化として安全に売れる」

### ターゲット外

| 種別 | 理由 |
|------|------|
| AIチャットを社員に使わせているだけの会社 | 実行系ツールがないため不要 |
| AI調査・回答生成のみ | 外部アクションがないため不要 |
| 大企業全体のAIガバナンス部門 | 初期プロダクトではServiceNow/Microsoftに比べて範囲が狭い |
| 承認をすべて自社ワークフロー基盤で作れる大企業 | 初期ターゲットではない |

## 料金仮説

料金は実装前ヒアリングで検証する。現時点では、承認リクエスト量ではなく「本番導入リスクを下げる価値」に基づいて設計する。

| プラン | 月額仮説 | 対象 | 主な価値 |
|--------|----------|------|----------|
| Starter | ¥15,000 | 個人開発、小規模SaaS | Slack承認、基本監査ログ、月500件 |
| Standard | ¥50,000 | AIスタートアップ、中小SaaS | 複数プロジェクト、承認ルール、月5,000件 |
| Pro | ¥150,000 | 複数AI機能を持つSaaS/SIer | 高度な監査、権限管理、限定Level 2 |
| Enterprise | 応相談 | 規制業種、大企業 | SLA、SSO、Proxy、個別統制 |

ヒアリングでは「月5万円なら払うか」だけでなく、「何があれば稟議が通るか」「事故を防げるなら年間いくらまで妥当か」を確認する。

## 実装前検証計画

### 10社ヒアリング

最初に実装ではなく、10社以上に以下を確認する。

- AIエージェントに実行させたいが止めている操作は何か
- すでに自前承認Bot、Slack確認、手動レビューを作ったか
- 事故が起きた場合の損害は金額、信用、工数でどれくらいか
- 承認者はエンジニアか、業務部門か、顧客企業の担当者か
- Slack、Teams、LINE WORKS、メールのどれが必須か
- 監査ログや権限分離がないと社内稟議・顧客導入が止まるか
- 月5万円なら払うか。払わないなら何が足りないか

### デモで確認すること

抽象説明では「便利そう」で終わるため、Slack通知とWeb監査ログだけの薄いデモを見せる。

| デモ要素 | 検証したい反応 |
|----------|----------------|
| Slack承認通知 | 承認者に必要な情報が足りるか |
| Web承認一覧 | 管理者が運用状況を見たいと思うか |
| 監査ログ | 稟議・顧客説明に使えると感じるか |
| LINE WORKS Waitlist | 国内企業で本当に必須チャネルか |
| Level 2モック | 「AIにAPIキーを渡さない」価値が伝わるか |

### Go/No-Go基準

| 判定 | 条件 |
|------|------|
| Go | 10社中3社以上が明確な本番課題を持ち、うち2社以上が有料PoCに前向き |
| Pivot | 承認UIより、監査ログ/権限分離/通知チャネルのどれかに強い需要が偏る |
| Stop | 「怖い」はあるが自前実装も支払い意思もなく、導入先が曖昧 |

## 開発・実行スケジュール

### Phase 0: 実装前検証

- 10社ヒアリング
- 薄いデモまたはクリック可能プロトタイプ
- LINE WORKS Waitlist
- MVP言語と最初の通知チャネル決定
- 有料PoC候補2社の獲得

### Phase 1: MVP

- SDK/Adapterは1言語に絞る
- Slack通知 + 承認/却下
- 基本ダッシュボード（承認待ち一覧・処理履歴・監査ログ）
- APIキーとプロジェクト管理
- fail-closed
- テストモード

### Phase 1.5: 国内導入強化

- LINE WORKSまたはTeamsのうち、需要が強い方を1つ実装
- 承認ルールUIの最小版
- 有料PoCから月額課金へ移行

### Phase 2: セキュリティ差別化

- 限定用途のクレデンシャル保管
- 例: SendGrid代理送信、Stripe支払い作成、CRM更新のいずれか
- 代理実行ログ
- 権限テンプレート

## 顧客獲得戦略

### Phase 0: 直接ヒアリング

- AIエージェント開発者、AI SaaS、SIer新規事業担当に個別連絡
- Zenn/Qiitaで「AIエージェントに実行権限を渡す前に決めるべき承認・監査設計」を発信
- AIエージェント勉強会で薄いデモを見せる
- LINE WORKS必須企業のWaitlistを集める

### Phase 1: Design Partner

- 無料ベータではなく、有料PoCまたは導入コミット付きDesign Partnerを優先する
- 3社の実運用ログから、承認項目、通知チャネル、ルール需要を決める
- サンプル実装はGitHubに出すが、SDKスター獲得を主目的にしない

### Phase 2: Product-led Growth

- セルフサインアップ
- テストモードで導入検証しやすくする
- 監査ログと権限分離を事例化する

## リスクと対策

| リスク | 対策 |
|--------|------|
| 承認SDK単体がコモディティ化する | 外向け訴求を「実行権限管理・監査・権限分離」に寄せる |
| HumanLayer同様に承認単体が伸びない | 10社ヒアリングで支払い意思と既存自前実装を確認してから作る |
| 大手が承認機能を追加する | カスタムAIエージェント、日本企業向け通知、軽量導入、Level 2限定用途に絞る |
| LINE WORKSを作ったが使われない | Waitlistとヒアリングで需要確認後に実装する |
| Level 2実装が重くなる | 汎用Vaultにせず、1用途の代理実行から始める |
| 顧客がSDK導入を面倒がる | LangGraph/OpenAI Agents SDKのサンプルAdapterを用意する |
