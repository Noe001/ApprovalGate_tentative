# 稟議ゲート（仮） — AIエージェント実行権限管理インフラ

## エグゼクティブサマリー

AIエージェントは、メール送信、CRM更新、支払い、広告出稿、データ変更など、現実の業務を実行する段階に入りつつある。一方で、企業が本番導入で止まりやすい理由は「AIが何を言うか」ではなく、**AIにどこまで実行権限を渡してよいか**である。

**稟議ゲート（仮）**は、AIエージェントの危険な実行を、人間承認・監査ログ・権限分離で本番運用できるようにする実行権限管理インフラ。SDKは導入手段の1つであり、差別化の本体は「承認結果をエージェントへ戻す運用ループ」「ビジネスユーザーが使える承認UI」「監査証跡」「将来的なクレデンシャル分離」にある。

## 解決する問題

### AIエージェントの本番化を止めるのは「実行権限」

```
事故例:
・AIが顧客1,000人に誤ったキャンペーンメールを一斉送信
・広告出稿AIがループして高額な広告費を積み上げ
・「古いファイルを整理して」という指示で重要データを削除
・CRM更新AIが誤った条件で商談ステータスを書き換え
```

問題の本質は「AIが間違えること」だけではない。企業側に、危険なアクションだけを止め、担当者が判断し、結果をエージェントに戻し、後から説明できる仕組みがないことが問題である。

### 各社が抱える現実

| 現状 | 問題 |
|------|------|
| 承認フローを自前でSlack botとして実装 | 開発に2〜3週間かかり、例外処理・監査ログ・権限管理のメンテが残る |
| AI自動実行を封印している | 本来の業務効率化価値が出せない |
| 口頭で「人間が確認します」と説明 | 社内稟議・顧客説明で技術的担保にならない |
| AIに外部APIキーを渡している | 承認デコレータを外せば迂回でき、構造的な安全性がない |

## プロダクト概要

### 仕組み

```
【稟議ゲート（仮）なし】
AIエージェント → 外部API/業務システムを直接実行

【稟議ゲート（仮）あり Level 1】
AIエージェント → SDK/Adapter → 承認待ち
                         ↓
                 Slack/Webに通知
                         ↓
                 担当者が承認/却下
                         ↓
                 エージェントが再開/停止

【稟議ゲート（仮）あり Level 2】
AIエージェント → 稟議ゲート（仮）に実行依頼
                         ↓
                 承認後、稟議ゲート（仮）が外部APIを代理実行
                         ↓
                 AIエージェントは本物のAPIキーを持たない
```

### 導入コードイメージ

```python
from ringi_gate import approval_required

@approval_required(reason="メール送信", metadata={"recipient_count": 234})
def send_email(to, subject, body):
    actually_send_email(to, subject, body)
```

このコードはMVPの導入導線であり、最終的な価値は「この承認が誰に届き、どう判断され、どのログとして残り、どの実行権限をAIから分離できるか」にある。

## ターゲット顧客

**AIを使ったサービスや社内エージェントを本番運用したい開発チーム**を初期ターゲットにする。単にChatGPTを社員に使わせる企業ではなく、AIが外部にアクションする場面を持つチームに絞る。

### 理想顧客の条件

- AIエージェントにメール送信、CRM更新、支払い、データ変更などの実行系ツールを持たせている
- 小規模チームで、承認UI・監査ログ・例外処理を自前で作る余裕がない
- 社内稟議や顧客説明で「危険な操作は人間が止められる」と示す必要がある
- Slack、Teams、LINE WORKS、メールなど、業務担当者が日常的に使う通知導線が重要

## 競合認識

「AIエージェントを一時停止して承認する」機構そのものは、すでに差別化しにくい。

| 競合/代替 | 脅威 | 稟議ゲート（仮）の勝ち筋 |
|-----------|------|------------------|
| LangGraph / LangChain | interrupt/pause/resume がネイティブ化 | 承認UI、監査ログ、業務ルール、通知運用を提供する |
| OpenAI Agents SDK | tool approval がネイティブ化 | 特定SDKに閉じない運用・監査・権限分離レイヤーになる |
| HumanLayer | Python/TypeScript、Slack/SMS/emailの承認ワークフロー実績 | ピボットを「承認単体は弱い」警戒信号として捉え、実行権限管理へ寄せる |
| Microsoft Copilot Studio / Power Automate | Microsoft圏のHuman in the loopと承認が強い | カスタムAIエージェント、スタートアップ、国内SaaS、Microsoft外の開発チームに絞る |
| ServiceNow AI Control Tower | 大企業向けAIガバナンスが強い | 小規模〜中堅チームが軽く導入できる本番化ゲートに絞る |

## セキュリティ設計方針

SDK デコレータだけでは「付け忘れれば素通り」という構造的弱点がある。MVPでは導入しやすさを優先するが、事業上の差別化はLevel 2以降に置く。

| レベル | 内容 | 位置づけ |
|--------|------|----------|
| Level 1 | SDK/Adapter + fail-closed + 監査ログ | MVP。導入容易性と運用可視化を検証する |
| Level 2 | 稟議ゲート（仮）が外部APIキーを保管し、承認後に代理実行 | 差別化の中核。最初はメール送信など1用途に限定する |
| Level 3 | Outbound Proxyで全通信を強制経由 | エンタープライズ向け将来オプション |

## 実装前にやること

次にやるべき主タスクはフル実装ではなく、**10社ヒアリングと極小デモによる検証**である。

- 10社以上に「止めたいAI実行」「既存の自前承認」「支払い意思」を確認する
- Slack通知とWeb監査ログだけの薄いデモを作り、抽象論ではなく具体画面で反応を見る
- LINE WORKSは実装せず、Waitlistで需要を測る
- Level 2は汎用Vaultではなく、1用途の構造的保証として設計する
