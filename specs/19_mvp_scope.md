# MVPスコープ定義

## 目的

Phase 1 MVPは、稟議ゲート（仮）の価値を「承認SDK」ではなく、AIエージェントの危険な実行を本番運用するための承認・監査・権限管理として検証するための最小プロダクトである。

MVPの目的は機能を広く揃えることではない。以下の3つを証明する。

1. AIエージェントの危険な実行を止めたい顧客が存在する
2. Slack/Webの承認・監査ログだけでも本番導入の不安が下がる
3. 月額課金または有料PoCに進むだけの痛みがある

---

## フェーズ定義

| フェーズ | 名前 | 実装可否 | 目的 |
|----------|------|----------|------|
| Phase 0 | 実装前検証 | 本格実装しない | 10社ヒアリング、薄いデモ、支払い意思確認 |
| Phase 1 | MVP | 実装する | Slack承認、Web監査ログ、1言語SDK/Adapterで価値検証 |
| Phase 1.5 | 国内導入強化 | 条件付き実装 | LINE WORKSまたはTeamsを需要に応じて追加 |
| Phase 2 | 権限分離 | 条件付き実装 | 限定用途のLevel 2クレデンシャル保管 |
| Phase 3 | Enterprise | 後回し | 複数承認、Proxy、SLA、コンプライアンス |

---

## Phase 1 MVPに含めるもの

### 必須機能

| 機能 | MVP範囲 | 成功条件 |
|------|---------|----------|
| 1言語SDK/Adapter | PythonまたはTypeScriptの片方。Phase 0で決定 | Design Partnerが実アプリに接続できる |
| 承認リクエスト作成API | `reason`, `description`, `metadata`, `timeout` を受け取る | 危険な実行がPENDINGとして記録される |
| Slack通知 | 承認/却下ボタン、主要メタデータ、詳細リンク | 承認者がSlackだけで判断できる |
| Web承認一覧 | PENDING/APPROVED/REJECTEDの一覧・詳細 | 管理者が運用状況を見られる |
| 監査ログ | request.created / approved / rejected / timed_out | 社内稟議・顧客説明に使える |
| APIキー/プロジェクト | 1テナント複数プロジェクトまで | 顧客の複数エージェントを分けられる |
| fail-closed | API障害時は拒否扱い | 危険な実行が素通りしない |
| テストモード | 通知なしでリクエスト記録・手動承認 | ステージング/CIで試せる |

### 最小UI

| 画面 | MVP範囲 |
|------|---------|
| ダッシュボード | 承認待ち件数、直近リクエスト、エラーだけ |
| 承認一覧 | フィルターはステータスとプロジェクトのみ |
| 承認詳細 | metadata表示、承認/却下、監査ログ |
| プロジェクト設定 | APIキー発行、デフォルト承認者、タイムアウト |
| 通知設定 | Slack連携のみ。LINE WORKS/TeamsはWaitlist表示 |

---

## Phase 1 MVPに含めないもの

| 除外項目 | 理由 | 再検討条件 |
|----------|------|------------|
| LINE WORKS本実装 | 実装量が大きく、初期需要が未検証 | 10社中3社以上が必須と回答 |
| Teams本実装 | Microsoft圏需要が未検証 | LINE WORKSよりTeams需要が強い |
| 複数承認AND | 初期運用で過剰になりやすい | Design Partnerの本番稟議で必須 |
| エスカレーション | 例外パターンが多く重い | 承認遅延が明確な課題になる |
| モバイルアプリ | Slack/Webで代替可能 | 承認者がスマホ中心でSlack不十分 |
| 汎用承認ルールUI | 仕様が膨らみやすい | 手動承認数が多すぎる |
| Level 2汎用Vault | 技術・責任範囲が重い | 1用途の有料PoCが成立 |
| Outbound Proxy | インフラ変更が必要 | Enterprise案件が出た後 |
| SLA/SSO/SOC2 | 初期顧客には重い | Enterprise契約前 |

---

## MVP言語の決定ルール

Phase 0の10社ヒアリングで、以下の優先順位で最初のSDK/Adapter言語を決める。

1. 有料PoCに最も近いDesign Partnerが使う言語
2. 危険な実行アクションがすでに動いているシステムの言語
3. LangGraph/LangChain中心ならPython、OpenAI Agents SDK/Nodeサービス中心ならTypeScript
4. 同率ならTypeScriptを優先し、PythonはサンプルAPI連携で補う

## MVP成功指標

| 指標 | 目標 |
|------|------|
| ヒアリング完了 | 10社以上 |
| 明確な本番課題 | 3社以上 |
| 有料PoC候補 | 2社以上 |
| 実アプリ接続 | 1社以上 |
| 月額5万円以上の支払い意思 | 2社以上 |
| LINE WORKS必須回答 | 3社以上ならPhase 1.5候補 |
| Level 2に強い反応 | 2社以上ならPhase 2設計開始 |

---

## MVP後の判断

| 結果 | 判断 |
|------|------|
| Slack承認と監査ログだけで有料PoCが進む | Phase 1を製品化する |
| 承認UIより監査ログが刺さる | 監査・稟議資料生成へ寄せる |
| LINE WORKSが必須 | Phase 1.5でLINE WORKSを優先する |
| Teamsが必須 | Phase 1.5でTeamsを優先する |
| APIキー分離が一番刺さる | Phase 2の限定Level 2を前倒しする |
| 支払い意思が弱い | 承認ワークフロー単体からピボットする |
