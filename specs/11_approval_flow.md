# 承認フロー詳細設計

## リクエストのライフサイクル

### 状態遷移図

```
                    SDK/Adapter/APIからリクエスト受信
                           │
                    ┌──────▼──────┐
                    │   PENDING    │  承認待ち
                    │   (承認待ち)  │
                    └──────┬──────┘
                           │
               ┌───────────┼───────────────┐
               │           │               │
        ルール評価       人間の操作        タイムアウト
               │           │               │
      ┌────────▼───┐    ┌──┴──┐     ┌──────▼──────┐
      │ AUTO_       │    │     │     │  TIMED_OUT   │
      │ APPROVED    │    ▼     ▼     │ (タイムアウト) │
      │ (自動承認)  │ APPROVED REJECTED└──────┬──────┘
      └────────────┘ (承認済み)(却下)         │
                                    タイムアウト時の
                                    デフォルト動作
                                    ├── 拒否 → REJECTED
                                    └── 承認 → APPROVED
```

### 状態一覧

| 状態 | 値 | 説明 |
|------|-----|------|
| 承認待ち | `PENDING` | SDK/Adapter/APIからリクエストを受信し、人間（またはルール）の判断待ち |
| 承認済み | `APPROVED` | 人間またはルールにより承認された |
| 却下 | `REJECTED` | 人間またはルールにより却下された |
| 自動承認 | `AUTO_APPROVED` | ルールにより自動的に承認された（人間の関与なし） |
| タイムアウト | `TIMED_OUT` | 指定時間内に判断がなく、タイムアウト処理が適用された |
| エラー | `ERROR` | システムエラーにより処理できなかった |

`ERROR` 状態の回復フロー（手動で「再処理する」ボタンを押して `PENDING` に戻す手順）は `16_system_behavior.md` セクション4「ERROR 状態の回復」を参照。

## リクエスト受信から通知までの処理フロー

```
① SDK/Adapter/API連携がAPIエンドポイントにリクエストを送信
           ↓
② APIキーの検証（無効なら即時 401 エラー）
           ↓
③ リクエストをDBに PENDING 状態で記録
           ↓
④ ルールエンジンを実行
   ├── 一致するルールあり
  │    ├── アクション: 自動承認 → AUTO_APPROVED に更新 → SDK/Adapter/APIにOK応答
  │    ├── アクション: 自動却下 → REJECTED に更新 → SDK/Adapter/APIに却下応答
   │    └── アクション: 指定承認者に通知 → ⑤へ
   └── 一致するルールなし → デフォルト承認者に通知 → ⑤へ
           ↓
⑤ 通知を送信
  ├── Slack（Phase 1）
  └── LINE WORKS / Teams（Phase 1.5候補。検証後に実装）
           ↓
⑥ SDK/Adapter/API連携はポーリングまたはWebhookで結果を待つ
           ↓
⑦ 人間が承認・却下 → APPROVED / REJECTED に更新
  SDK/Adapter/API連携にコールバック（Webhook）またはポーリング応答
           ↓
⑧ タイムアウト時間が経過した場合（未処理のまま）
  → TIMED_OUT に更新 → タイムアウト設定の動作（拒否/承認）をSDK/Adapter/API連携に返す
```

---

## SDK/Adapter/API連携との通信方式

### 同期ポーリング（Polling）

SDK/Adapterが一定間隔でAPIをポーリングし、結果を取得する方式。

- SDK/Adapterがリクエスト送信後、リクエストIDを取得
- 1秒ごとに `/requests/:id/status` をポーリング
- 最大ポーリング時間: タイムアウト時間 + 30秒（バッファ）
- タイムアウト後もSDKが待機している場合は、タイムアウト結果を返す

### Webhookコールバック（将来対応）

あらかじめ登録されたURLに、承認・却下時にHTTP POSTを送信する方式。

- SDK/Adapterが起動中でなくても結果を受け取れる
- バッチ処理・非同期エージェントに対応
- Webhook URLはプロジェクト設定で登録

## fail-closed の動作

稟議ゲート（仮）のAPIサービスが利用不能な場合（タイムアウト・接続エラー等）:

```
デフォルト動作:
  SDK/Adapterはすべてのアクションを「拒否」として扱う
  エラーの詳細をログに記録
  AIエージェントには PermissionDeniedError を発生させる
```

- fail-closedはテナント設定でOFF（fail-open）に変更可能
- fail-openに設定した場合は管理画面に警告バナーを表示

## タイムアウト処理の詳細

### タイムアウトまでの流れ

| 経過時間 | 動作 |
|---------|------|
| 0分 | リクエスト受信・通知送信 |
| タイムアウト時間の75% | 「もうすぐタイムアウトします」を承認者にリマインド通知 |
| タイムアウト時間 | TIMED_OUT に更新・タイムアウト設定の動作をSDK/Adapter/API連携に返す |

### タイムアウト時間の優先順位

```
ルール設定のタイムアウト時間（最優先）
  ↓ なければ
プロジェクト設定のタイムアウト時間
  ↓ なければ
テナント設定のデフォルトタイムアウト時間（デフォルト: 30分）
```

## エスカレーション

### 自動エスカレーション

ルール設定でエスカレーションアクションを設定した場合:

1. リクエスト受信時にまず一次承認者に通知
2. 一次承認者が `{待機時間}` 分以内に応答しない場合、二次承認者に通知
3. 二次承認者にも通知が届くが、一次承認者も引き続き承認可能

### 手動エスカレーション

承認者が「自分では判断できない」と感じた場合、Web画面から別のメンバーに手動でエスカレーション可能。

- 承認者の権限で実行可能
- エスカレーション先のメンバーに通知が届く
- 元の承認者も引き続き承認可能

---

## 複数承認者の扱い

1つのリクエストに複数の承認者が設定されている場合:

| 設定 | 動作 |
|------|------|
| OR（いずれか1人）| いずれか1人が承認・却下すれば確定。他の承認者には「処理済み」通知 |
| AND（全員）| 全員が承認して初めて確定。1人でも却下すれば即座に却下確定。**1人が却下し残りがタイムアウトした場合も「却下」として確定**（却下は承認より優先されるため）。全員がタイムアウトした場合はタイムアウト設定の動作を適用 |

デフォルトはOR。プロジェクト設定で変更可能。

## 監査証跡

すべての状態遷移は変更不可の監査ログに記録される。

| 記録項目 |
|---------|
| リクエストID |
| 操作種別（承認/却下/自動承認/タイムアウト/エスカレーション） |
| 実行者（ユーザーID または "system"） |
| 実行日時（UTC、ミリ秒精度） |
| 実行場所（Web / Slack / API。LINE WORKS/TeamsはPhase 1.5以降） |
| IPアドレス（Webからの操作の場合） |
| 変更前・変更後のステータス |
