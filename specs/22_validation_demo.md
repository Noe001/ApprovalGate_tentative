# 検証用デモ仕様

## 目的

本格実装前のヒアリングで、稟議ゲート（仮）の価値を具体的に見せるための薄いデモを定義する。デモは完成品ではなく、顧客の反応を引き出すための会話道具である。

---

## デモの原則

- 実装量を最小化する
- Slack承認とWeb監査ログだけを見せる
- LINE WORKSは実装せず、Waitlistとして表示する
- Level 2は実API連携せず、図とモックログで価値を説明する
- デモ後に「使いたいか」ではなく「どの業務なら使うか」を聞く

---

## デモシナリオ

### シナリオ1: 一斉メール送信

AIマーケティングエージェントが、顧客234名にキャンペーンメールを送ろうとする。

| 項目 | 値 |
|------|-----|
| アクション | メール送信 |
| 理由 | 顧客へのキャンペーン通知 |
| 送信先件数 | 234 |
| リスク | 一斉送信 |
| 承認者 | マーケティング責任者 |
| デフォルト動作 | タイムアウト時は拒否 |

見せるもの:

1. Slack承認通知
2. Web承認詳細
3. 監査ログ
4. 却下時にAIへ理由が返る流れ

### シナリオ2: CRM更新

AI営業エージェントが、商談ステータスを一括更新しようとする。

| 項目 | 値 |
|------|-----|
| アクション | CRM一括更新 |
| 理由 | 休眠リードのステータス整理 |
| 対象件数 | 86 |
| リスク | 顧客データ変更 |
| 承認者 | 営業Ops |
| デフォルト動作 | タイムアウト時は拒否 |

見せるもの:

1. metadataが承認判断に必要か
2. 監査ログが顧客説明に使えるか
3. ルール自動化が必要か

### シナリオ3: Level 2モック

AIエージェントがSendGrid APIキーを持たず、稟議ゲート（仮）に送信依頼だけを出す。

見せるもの:

```
AIエージェント → 稟議ゲート（仮）に送信依頼
    ↓
人間が承認
    ↓
稟議ゲート（仮）がSendGridキーで代理送信
    ↓
AIエージェントにはAPIキーを渡さない
```

確認したい反応:

- APIキーをAIに渡さないことに価値を感じるか
- 最初に代理実行してほしい外部APIは何か
- これがあると稟議や顧客説明が通りやすいか

## LINE WORKS Waitlist表示

```text
LINE WORKS連携は検証中です

Slackを利用していないチーム向けに、LINE WORKS承認通知を検討しています。
必要な場合は利用希望を登録してください。

[利用希望を登録する]
```

聞くこと:

- LINE WORKSがないと導入できないか
- TeamsとLINE WORKSならどちらが優先か
- 承認者は個人DMでよいか、グループ/トークルームが必要か

## デモ後の必須質問

1. このデモを自社のどのAIエージェントに使えそうですか
2. Slack通知の情報だけで承認判断できますか
3. 監査ログは稟議や顧客説明に使えますか
4. LINE WORKS/Teamsは必須ですか
5. AIにAPIキーを渡さないLevel 2はどれくらい重要ですか
6. 月5万円なら導入検討できますか
7. 有料PoCなら何が必要ですか

---

## デモの合格基準

- 10社中3社以上が具体的な利用業務を挙げる
- 2社以上が監査ログを稟議/顧客説明に使えると言う
- 2社以上が有料PoCに前向き
- LINE WORKS/Teamsの優先順位が判断できる
- Level 2の最初の対象API候補が1つに絞れる
