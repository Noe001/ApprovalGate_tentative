# 通知仕様

## 通知チャネル一覧

| チャネル | 用途 | 設定場所 |
|---------|------|---------|
| Slack | Phase 1 MVPの主要通知先。ボタンで承認・却下が完結 | テナント全体設定 + 個人設定 |
| LINE WORKS | Phase 1.5候補。Slack非採用の国内企業向け。Waitlist/ヒアリングで需要確認後に実装 | テナント全体設定 + 個人設定 |
| Microsoft Teams | Phase 1.5候補。Microsoft 365環境が多い場合にLINE WORKSと優先度を比較 | テナント全体設定 + 個人設定 |
| メール | 補助通知。即時またはまとめ配信 | 個人設定 |
| Webプッシュ | ブラウザ通知（将来対応） | 個人設定 |

## Slack通知

### 通知のトリガー

| トリガー | 通知先 |
|---------|--------|
| 承認リクエスト受信（人間承認が必要な場合） | 担当承認者 |
| タイムアウト時間の75%経過（リマインド） | 担当承認者 |
| タイムアウト発生 | 担当承認者 + 管理者（設定による） |
| エスカレーション受信 | エスカレーション先の承認者 |
| 自分が担当するリクエストが他者に承認された | 自分（通知設定による） |

### 承認リクエスト通知のメッセージ構成

```
┌────────────────────────────────────────────────────────┐
│ 🔔 稟議ゲート（仮）                                            │
│                                                        │
│ *承認リクエストが届きました*                               │
│                                                        │
│ *アクション:* メール送信                                  │
│ *理由:* 顧客へのキャンペーン通知                           │
│ *プロジェクト:* marketing-agent                         │
│                                                        │
│ ─── 詳細 ────────────────────────────────             │
│ 宛先: customers@list.csv                               │
│ 件名: 新機能リリースのご案内                               │
│ 受信者数: 234名                                         │
│                                                        │
│ ⏱ タイムアウトまで 30分                                  │
│                                                        │
│ [✅ 承認する]  [❌ 却下する]  [🔗 詳細を見る]              │
└────────────────────────────────────────────────────────┘
```

### メッセージ要素の詳細

| 要素 | 内容 |
|------|------|
| アクション名 | SDK/Adapter/APIの `reason` パラメータ |
| 理由 | SDK/Adapter/APIの `description` パラメータ（設定されている場合） |
| プロジェクト名 | リクエスト元のプロジェクト |
| 詳細（メタデータ） | SDK/Adapter/APIの `metadata` パラメータのキー/値を最大5件表示。超える場合は「他X件」 |
| タイムアウトカウントダウン | 設定されている場合のみ表示 |
| 承認するボタン | クリックで即時承認。確認なし |
| 却下するボタン | クリックでSlack上にモーダル（Block Kit modal）が開き、却下理由を入力できる |
| 詳細を見るリンク | Web上の承認リクエスト詳細ページに遷移 |

### 承認・却下後のメッセージ更新

承認または却下後、元のメッセージが更新される（ボタンが消え、結果が表示される）:

```
✅ 田中 太郎 が承認しました（2分後）
```

### Slack Interactivity セキュリティ

Slackのインタラクティブコンポーネント（ボタンクリック）はSlackからHTTP POSTで送られる。このリクエストの正当性を検証するため:

- すべてのSlack Interactivityエンドポイントで `X-Slack-Signature` ヘッダーをHMAC-SHA256で検証する
- `X-Slack-Request-Timestamp` と `SLACK_SIGNING_SECRET` を使用して署名を検証
- タイムスタンプが5分以上古い場合はリクエストを拒否（リプレイアタック防止）

### 通知の失敗とリトライ

Slack通知の送信に失敗した場合（チャンネル未見つかり・権限エラー等）:

- 最大3回まで指数バックオフでリトライ（1秒 → 4秒 → 16秒）
- リトライ後も失敗した場合は承認リクエストの処理履歴に「通知失敗」として記録
- 通知失敗はAdmin+に向けたシステム通知に含める（即時ではなくまとめ報告）

### リマインド通知

タイムアウトまで残り25%（デフォルト: 30分設定なら7.5分前）になったとき:

```
⏰ まもなくタイムアウトします

「メール送信」のリクエストが *残り8分* でタイムアウトします。
タイムアウト時は自動的に拒否されます。

[✅ 承認する]  [❌ 却下する]  [🔗 詳細を見る]
```

## LINE WORKS通知（Phase 1.5候補）

LINE WORKSは国内中堅企業への重要な入口になり得るが、Phase 1 MVPでは実装しない。Phase 0のヒアリングとWaitlistで、10社中3社以上が必須チャネルとして挙げた場合に優先実装する。

Slack通知と同等の内容をLINE WORKSのBot経由で送信。

### 差異点

- ボタンUIはLINE WORKSのFlexメッセージ形式で実装
- 却下時の理由入力はSlackのモーダルではなく、テキストリプライで入力する形式
  - Botが「却下理由を返信してください」と促す
  - ユーザーがテキストを返信すると却下処理が完了
- メッセージ形式はLINE WORKSのAPIに準拠

## メール通知

### 即時メール

承認リクエスト受信時に即時送信（個人設定で即時を選んでいる場合）。

**件名:** `[承認待ち] メール送信 – marketing-agent`

**本文構成:**
1. アクション名・理由
2. メタデータ（主要項目）
3. タイムアウト日時（UTCではなくユーザーのタイムゾーンで表示）
4. 「Webで承認・却下する」リンク（メールからの直接承認ボタンはセキュリティ上提供しない。必ずWebダッシュボードまたはSlackで操作する）

### まとめメール（1時間ごと / 日次）

件数が多い場合に1通にまとめて送信。

**件名（1時間まとめ）:** `[承認待ち 3件] 未処理の承認リクエストがあります`

**本文:**
- 未処理の承認待ちリクエスト一覧
- 各リクエストへのリンク
- 「ダッシュボードで一括確認する」リンク

## 管理者向けシステム通知

### サービス障害通知

稟議ゲート（仮）のAPI障害やエラー率の上昇時に管理者（Admin+）に通知。

| 条件 | 通知内容 |
|------|---------|
| APIエラー率が5%を超えた | 「サービスの一部に問題が発生しています」 |
| fail-closedが発動した | 「サービス障害によりリクエストを拒否しました」 |
| Slack連携が切れた | 「Slack連携が無効になりました。再設定してください」 |

### リクエスト量の急増通知

通常の1時間あたり平均の3倍以上のリクエストが来た場合:

```
⚠️ リクエスト量が急増しています

marketing-agentから過去1時間に通常の5倍（250件）のリクエストが届いています。
AIエージェントが意図しないループを起こしている可能性があります。

[リクエストを確認する →]
```

## 通知の重複・フラッドコントロール

- 同一リクエストへの通知は1チャンネルにつき1回のみ（リマインドを除く）
- 1分間に同一プロジェクトから10件以上のリクエストが来た場合: 個別通知をバッチ通知に切り替え
- バッチ通知: 「marketing-agentから5件の承認リクエストが届いています。[まとめて確認する]」
