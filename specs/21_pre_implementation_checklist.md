# 実装開始前チェックリスト

## 目的

フル実装を始める前に、稟議ゲート（仮）が「作れば使われる」状態に近いかを確認する。以下を満たすまで、Phase 1 MVPの本格実装には入らない。

---

## 必須完了条件

### 顧客検証

- [ ] 10社以上にヒアリングした
- [ ] 3社以上が明確なAIエージェント本番化課題を持っている
- [ ] 2社以上が有料PoCまたは導入コミットに前向き
- [ ] 1社以上が実アプリ接続に協力できる
- [ ] 月5万円以上の価格に対して、2社以上から強い拒否がない

### 問題の明確化

- [ ] 最初に止めるべき危険操作が1〜2種類に絞れている
- [ ] 承認者のペルソナが明確になっている
- [ ] 事故時の損害や稟議上の痛みが言語化できている
- [ ] 「便利そう」ではなく「ないと本番に進みにくい」理由がある

### MVPスコープ

- [ ] 最初のSDK/Adapter言語を決めた
- [ ] SlackだけでPhase 1を始める判断ができている
- [ ] LINE WORKS/Teamsの優先順位をヒアリング結果で判断した
- [ ] Phase 1に含めない機能を合意した
- [ ] Level 2の最初の候補ユースケースを1つ選んだ

### デモ検証

- [ ] Slack承認通知のサンプルを見せた
- [ ] Web承認一覧/監査ログのサンプルを見せた
- [ ] LINE WORKS Waitlistの反応を確認した
- [ ] Level 2の価値を図またはモックで説明した
- [ ] デモ後に「いつ使えるか」または「試したい」という反応があった

### 販売・導入

- [ ] Design Partner候補を2社以上リスト化した
- [ ] 導入決裁者と利用者の違いを確認した
- [ ] 有料PoCの提案条件を決めた
- [ ] 価格仮説を更新した
- [ ] 初期導入に必要なセキュリティ資料を洗い出した

---

## Go / Pivot / Stop 判定

### Go

以下をすべて満たす場合、Phase 1 MVP実装に進む。

- 10社中3社以上が明確な本番課題を持つ
- 2社以上が有料PoCに前向き
- 最初の危険操作が1〜2種類に絞れている
- Slack/Web監査ログだけでも試す価値があると言われている

### Pivot

以下の場合、実装前に訴求やスコープを変える。

| 観察 | ピボット先 |
|------|------------|
| 承認UIより監査ログに反応が強い | AI実行監査ログ/稟議証跡SaaS |
| 通知チャネルが最大課題 | LINE WORKS/Teams中心の国内AI承認ゲート |
| APIキー分離が最も刺さる | 限定用途のLevel 2代理実行から開始 |
| SIerだけが強く反応 | SIer向け顧客説明・テンプレート同梱ツール |

### Stop

以下の場合、承認ワークフロー単体の実装は止める。

- 自前承認を作っている会社がほぼない
- 「怖い」と言うが予算も導入責任者もない
- Slack/Teams/LINE WORKSなど既存ワークフローで十分と言われる
- 月額課金の支払い意思が確認できない

---

## 実装開始時の最小バックログ

Go判定後、最初に作るバックログは以下に限定する。

1. テナント/プロジェクト/APIキー
2. 承認リクエスト作成API
3. Slack通知と承認/却下エンドポイント
4. 承認一覧/詳細/監査ログ
5. 1言語SDK/Adapter
6. fail-closed
7. テストモード

これ以外の機能は、Design Partnerの実運用ログを見てから追加する。
