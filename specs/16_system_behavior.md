# システム横断的な動作仕様

## 1. タイムゾーン方針

### 基本ルール

**すべての時刻関連機能はテナント設定のタイムゾーンを使用する。**

- `10_pages_settings.md` で設定するテナントタイムゾーン（デフォルト: `Asia/Tokyo`）が全機能に適用される
- DBへの保存は常にUTC
- 表示・比較・ルール評価・通知はテナントのタイムゾーンに変換して処理する

### 影響を受ける機能

| 機能 | 適用内容 |
|------|---------|
| アナリティクス | 「今日」「過去7日間」などの期間区切りはテナントTZの0時 |
| 承認ルール（時刻条件） | `9:00〜18:00` はテナントTZでの時刻として解釈 |
| タイムアウトカウントダウン | 表示はテナントTZ。内部計算はUTC |
| 通知・日次まとめメール | 「日次」はテナントTZの0時に配信 |
| 監査ログの表示 | テナントTZに変換して表示（UTC値もツールチップで確認可能） |

### 夏時間（DST）の扱い

- 時刻条件ルール（例: 9時〜18時）はDSTを考慮したうえで評価する
- DSTが発生するタイムゾーン（America/New_York 等）では、システム内部で自動的に調整する
- 日本標準時（Asia/Tokyo）はDSTなしのため、デフォルト設定での影響は発生しない

## 2. 削除カスケード仕様

### ユーザー削除

| 影響データ | 処理 |
|-----------|------|
| テナントメンバー | 即時削除 |
| 担当中の承認待ちリクエスト | 下記「再割り当てロジック」に従い再割り当て |
| エスカレーション待機中のリクエスト | エスカレーションをキャンセルし、元の一次承認者に戻す |
| ルールのエスカレーション先に設定されている場合 | そのルールのエスカレーション先をプロジェクトのデフォルト承認者に自動変更。Admin+にアラート通知を送る |
| 完了済みリクエストの `decided_by_id` | NULL にせず保持（監査のため）。表示上は「削除済みユーザー」と表示 |
| 監査ログの `actor_id` | 保持。表示上は「削除済みユーザー」と表示 |
| ユーザーレコード | ソフトデリート（`deleted_at` を記録、30日後に物理削除） |

**再割り当てロジック（優先順位順）**

```
1. 対象プロジェクトのデフォルト承認者に再割り当て
2. プロジェクトにデフォルト承認者がいない場合 →
   テナントのデフォルト承認者に再割り当て
3. テナントにもデフォルト承認者がいない場合 →
   テナントの Owner に割り当て（Owner が複数いる場合は全員）
4. Owner もいない場合 → ERROR 状態に変更し全 Admin に緊急通知
```

### プロジェクト削除

| 影響データ | 処理 |
|-----------|------|
| 承認待ちリクエスト | 即時タイムアウト扱い（`TIMED_OUT`）→ タイムアウト設定の動作を適用 |
| 完了済みリクエスト | ソフトデリート（30日後に物理削除） |
| APIキー | 即時失効（物理削除） |
| ルール | 即時削除 |
| プロジェクトレコード | ソフトデリート（30日後に物理削除） |

### テナント削除

| 影響データ | 処理 |
|-----------|------|
| 全サブデータ（プロジェクト・ルール・リクエスト・メンバー） | ソフトデリート（30日後に物理削除） |
| Slack / LINE WORKS / Teams 連携 | 実装済みチャネルのみ即時解除。LINE WORKS/TeamsはPhase 1.5以降 |
| Stripe サブスクリプション | 即時キャンセル |

### ルール削除

| 影響データ | 処理 |
|-----------|------|
| 処理中（PENDING）のリクエスト | 影響なし。ルール評価はリクエスト受信時点で確定している |
| 将来のリクエスト | 削除されたルールは以後評価されない |

## 3. デフォルト承認者が未設定の場合の動作

### 発生条件

- リクエストが届いたが、どのルールにも一致しない
- かつプロジェクトにもテナントにもデフォルト承認者が設定されていない

### 動作

1. リクエストを `ERROR` 状態で記録
2. テナントの Owner と全 Admin にメール・Slack で緊急通知を送る
3. 通知内容: 「承認者が設定されていないため、リクエストを処理できませんでした。ルールまたはデフォルト承認者を設定してください。」
4. SDK には `ApprovalServiceException`（設定エラー）を返す

### 予防UI

プロジェクトを作成する際、デフォルト承認者が未設定で保存しようとすると警告バナーを表示:

```
⚠️ デフォルト承認者が設定されていません。
どのルールにも一致しないリクエストが承認されなくなります。
[承認者を設定する] または [スキップして後で設定する]
```

## 4. ERROR 状態の回復

`ERROR` 状態のリクエストは一般的に手動での対処が必要。

### ERROR になる条件

| 原因 | 例 |
|------|-----|
| 設定不備 | デフォルト承認者なし・ルールにもマッチしない |
| 内部エラー | DB書き込み失敗 |

### 回復フロー

1. Admin+ がダッシュボードで ERROR リクエストを確認
2. 詳細ページでエラー理由を確認
3. 設定を修正（デフォルト承認者を追加、等）
4. 「再処理する」ボタンをクリック → PENDING に戻し、再度ルール評価を実行
5. 回復不可能な場合は「手動で承認」または「手動で却下」を選択

## 5. テストモードの使い分け

SDK には2つの異なるテスト手段がある。用途によって使い分ける。

| 手段 | 設定方法 | 動作 | いつ使うか |
|------|---------|------|---------|
| **テストキー** | APIキー `rg_test_...` を使用 | DBに記録される・Slack等の通知は送られない・ダッシュボードのテストタブで確認・承認・却下ができる | ステージング環境や統合テストで、実際のフロー全体を確認したい時 |
| **モックモード** | 環境変数 `RINGI_GATE_MOCK_MODE=auto_approve` または `auto_deny` | APIへの通信なし・全アクション即時承認または即時却下 | ユニットテスト、CIパイプライン、ローカル開発で本物のAPIを使いたくない時 |

### テストリクエストの確認場所

テストキーを使ったリクエストは、ダッシュボードおよび承認リクエスト一覧ページに **「テスト」バッジ**が付いた状態で表示される。フィルター「テストのみ / 本番のみ / すべて」で切り替え可能。

## 6. エラーページ仕様

| シナリオ | 表示内容 |
|---------|---------|
| 存在しないURL (`404`) | 「お探しのページは見つかりませんでした」+ ダッシュボードへのリンク |
| 権限不足 (`403`) | **「Not Found」と同じ画面を表示**（リソースの存在を示さない）+ ダッシュボードへのリンク |
| サービス障害 (`503`) | 「稟議ゲート（仮）は現在メンテナンス中です。しばらくしてから再度アクセスしてください。」+ ステータスページへのリンク |
| 認証切れ | ログイン画面にリダイレクト（ログイン後は元のURLに戻る） |

### 403 を 404 として表示する理由

他テナントのリソースURL（`/t/other-tenant/projects/123`）にアクセスされた際、「アクセス禁止」と表示するとリソースの存在を間接的に示すことになる。セキュリティのため一律「Not Found」を返す。

## 7. APIキーローテーション手順

APIキーをダウンタイムなく安全に切り替える推奨手順。

1. プロジェクトのAPIキー管理画面で「+ 新しいキーを発行」
2. 新しいキーをコピー
3. 自社のシステムの環境変数に新しいキーをデプロイ（古いキーは残したまま）
4. デプロイ後、新しいキーでリクエストが正常に届いていることをダッシュボードで確認
5. 古いキーを「失効」させる

**重要**: 古いキーを失効させてからコードをデプロイすると、デプロイ中にリクエストが通らなくなる。必ず「新しいキーで動作確認」→「古いキーを失効」の順で行うこと。

## 8. 自己承認の禁止

AIエージェントのリクエストを、そのエージェントのAPIキーと同一ユーザーが承認することについての制限はない（AIエージェントとユーザーは異なるエンティティのため）。

ただし以下の状況ではシステムが警告を表示する:

- Admin が自分自身の操作を起点とするリクエスト（例: Admin自身がトリガーしたAIエージェントのリクエスト）を承認しようとした場合: 警告バナー表示（「あなた自身がトリガーしたリクエストです」）。ブロックはしないが履歴に記録する。

## 9. 優先度の整理（タイムアウト・ルール・デフォルト承認者）

### タイムアウト時間の優先順位（高い順）

```
1. デコレータのパラメータ（@approval_required の timeout 引数）
2. ルール設定の timeout_seconds
3. プロジェクト設定の timeout_seconds
4. テナント設定の default_timeout_seconds（デフォルト: 1800秒）
```

タイムアウトリマインド通知（75%経過）は、**上記の優先順位で最終決定されたタイムアウト時間**の 75% 経過時点で送信する。

### ルール評価の優先順位

- ルールはプロジェクト設定ルール → テナント設定ルール の順に評価される
- 各グループ内では `order` 番号が小さいルールが優先
- **最初に一致したルールのアクションが実行される（first-match wins）**
- 一致ルールが複数ある場合でも、2番目以降は評価されない

### デフォルト承認者の優先順位（ルール一致なし時）

```
1. ルール設定の承認者指定（ルールが一致した場合のみ）
2. プロジェクト設定のデフォルト承認者
3. テナント設定のデフォルト承認者
4. 上記すべて未設定 → ERROR 状態（セクション3参照）
```
